## 0.5 ID & numbering standards (LOCKED - follow in every flow)

| ID | Rule | Example |
|---|---|---|
| Numeric IDs (`dag_id`,`batch_id`,`job_id`,`proc_id`) | **DB IDENTITY - never hand-pick, never embed in names**; scripts look them up by name | `DECLARE @job INT = (SELECT job_id ... WHERE job_name = ...)` |
| `dag_uid` (project/flow id) | `df_{domain}_{subject}`, lowercase snake_case, **stable forever** | `df_audiology_referrals` |
| External project code | **`meta.dag.project_code`** (file 23) + `manifest.yml` - best practice even though not mandated; scaffolder: `--project-code` | `HA-2026-017` |
| `batch_name` | `{dag_uid}_b{n}` (globally UNIQUE) | `df_audiology_referrals_b1` |
| `job_name` | `{dag_uid}_{purpose}` (UNIQUE per batch) | `df_audiology_referrals_ingest` |
| `proc_name` | `verb_object` (UNIQUE per job) | `ingest_referral`, `recon_appointment` |
| `rule_uid` (purge) | `{dag_uid}-R{n}` | `df_audiology_referrals-R1` |
| **Order columns** (`batch_order`,`job_order`,`proc_order`) | **gap numbering: 10, 20, 30 ...** - insert later steps (15) without renumbering; **equal numbers run in PARALLEL** (§15.8) | silver=10, scd2=20, recon=30 |
| Where a flow's IDs are documented | its `manifest.yml` (the per-flow register) + these conventions | |

## 0.1 Manifest YAML gotcha - quote any value containing a comma

In the `columns:` list, **quote a column `type` that contains a comma** (for example
`type: "decimal(18,2)"`). In YAML flow style (`{name: amount, type: decimal(18,2)}`) an
unquoted comma is parsed as a field separator, so the type silently truncates to
`decimal(18`. The generator cannot detect this (it sees the already-broken value), so the
rule is: quote `decimal(p,s)`, `numeric(p,s)`, `varchar(n,...)` and anything else with a comma.
Verified by the 2026-06-15 P-R24 generator-vs-DB spike.

## Evolving a flow & the convergence standard (added 2026-06-12; mirrors dummys doc 14 §5.11)

**Adding a table to an existing flow:** same `dag_uid`/`project_code` when subject, source
system, schedule and owner match - append object + columns + watermark/lock init + a proc row
to the existing seeds and redeploy. New project only for a genuinely different subject/schedule.

**Changing existing values - the standard:**
- `meta.*` seeds are **CONVERGENT**: full `MERGE` (INSERT when missing, **UPDATE mutable
  columns when matched**). A guard-only `IF NOT EXISTS ... INSERT` silently ignores your edit
  when the row exists. Until the examples/scaffolder are upgraded to this shape (tracked
  follow-up), pair your edit with an idempotent UPDATE.
- `log.*` seeds (watermark/lock init) are **INITIALIZE-ONLY by design** - never convert them
  to MERGE: a redeploy must never reset a watermark (it would re-ingest everything).
- Keys never change in place (`dag_uid`, object identity); retire with `is_enabled = 0`,
  never DELETE.

**Staged onboarding (source not ready yet?):** objects are **born disabled** (default 0,
DDL 30) - seed and merge NOW, develop on the mock harness, then enable with a one-line
convergent UPDATE once `dataflows-run-tests (suite=connectivity)` is green for the env
(see AUD-002 seed 02 §5b for the exact shape). Never hardcode around the framework.
