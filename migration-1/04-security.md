# 04 - Security

Identity and groups, the serving access model (who touches tables vs views, who sees CLS/RLS), RLS/CLS/DDM
mechanics, PHI by layer, and the legal/clinical obligations (de-identification, retention, breach).

## 1. Identity, groups, Key Vault
- Domains: Clinical, Finance, Operations, HR, Audiology, Dynamics (+ Platform/shared).
- Groups: `sg-healthent-{domain}-{role}` (eng / read / admin / analyst). SPs: `sp-healthent-fabric-{env}`,
  `-devops`, `-purview`. All identity is Terraform-managed and re-applied on every rebuild.
- One Key Vault per domain per env; access by group/SP only; cross-domain access denied (expect 403, tested).
- No secrets in code - KV references only. Secret rotation alerts at 30/7 days before expiry.
- Conditional Access: MFA for all Fabric access. Prod PHI access is JIT/PIM for all domains; emergency access via
  a logged, time-boxed, post-reviewed **break-glass** account.

## 2. Serving access model (who gets what)
| Principal | Gold TABLES `[domain]` | `[domain]_TECH_VIEW` (full) | `[domain]_VIEW` (PHI omitted) | Sees PHI (CLS) | Row scope (RLS) |
|---|---|---|---|---|---|
| Framework pipeline (MI/SP) | read+write (sole writer) | reads | - | yes | all |
| Power BI semantic model (Direct Lake) | read gold tables (OneLake role) | - | - | model OLS | model RLS |
| Admin / break-glass | PIM only (audited) | yes (PIM) | yes | yes | all (audited) |
| Data engineers / developers | none standing (prod); dev via CI | YES | yes | yes | per policy |
| Business users / analysts | none | none | YES | no (omitted) | scoped to their rows |
| Report viewers (Power BI app) | none | none | consume the model | model OLS | model RLS |

- **`view` = business; `tech_view` = developers + pipeline.** Granted at SCHEMA level by Entra group; the table
  schema is never granted to consumers.
- **CLS = which view you get:** `[domain]_VIEW` omits PHI; `[domain]_TECH_VIEW` shows it.
- **RLS = a security policy on both views** keyed to group membership; business rows-scoped, eng/admin per policy.
- **Direct Lake path:** Power BI reads the tables, bypassing views -> CLS = model **OLS**, RLS = model **RLS**,
  same Entra groups. Security is NEVER at the dashboard.

## 3. RLS / CLS / DDM mechanics (GA on the lakehouse SQL endpoint)
- **RLS:** `CREATE SECURITY POLICY` + an inline TVF predicate. Default = **group-only** (IS_MEMBER of the domain
  read/analyst/eng/admin groups), auto-generated by `gen_views`, applied to both views. Per-row scoping
  (region/clinic) = the mapping-table predicate variant, only where a flow needs it.
- **CLS:** **column omission** in the consumption view (strongest; and the only option on a read-only lakehouse
  SQL endpoint). DDM (`ALTER TABLE ADD MASKED`) only in the Warehouse exception.
- Grants by Entra group, never individual users; tech/UNMASK behind PIM/JIT.

## 4. PHI by layer (restricted below Gold too)
- Bronze and Silver hold raw/identifiable PHI - readable by the framework MI + named break-glass ONLY, never
  general domain readers. Classify + sensitivity-label Bronze/Silver like Gold. Masking at Gold is not the whole
  control; least privilege applies below Gold.
- No raw PHI outside Prod - Dev/Test/UAT use synthetic or de-identified data only (a masked view is not
  de-identification). No pipeline/shortcut/export/backup path may move Prod Bronze/Silver to a lower env.
- Encryption-at-rest key model (CMK vs MMK) is an explicit decision per engagement (confirm current Fabric/OneLake
  CMK capability; record the choice + a review trigger).

## 5. Legal / clinical obligations (surface early; Privacy Officer/Legal sign-off)
- **De-identification standard:** define against OAIC guidance; audiology re-identification risk is real
  (audiogram patterns, rare diagnoses, small clinic + DOB/postcode). State technique, who may re-identify, how the
  mapping is protected, residual-risk acceptance. This is the approved source for non-prod data + any ML features.
- **Retention & erasure:** state-based minimum retention (commonly ~7y adults; minors to ~25 - confirm per
  jurisdiction), then APP 11.2 destruction/de-identification. Trap: Delta time-travel + OneLake retention keep
  "deleted" PHI - a soft-delete/DELETE does NOT satisfy erasure; honour via VACUUM / retention-window expiry /
  physical removal on Bronze, Silver, Gold, AND the control plane, with evidence. Encode windows in `meta.purge_rule`.
- **Notifiable Data Breaches (NDB, Part IIIC):** a standing process (not improvised): detect -> eligible-breach
  assessment within 30 days -> contain -> if likely serious harm, notify OAIC + individuals -> record-keeping.
  This is SEPARATE from technical incident MTTR (containment != statutory notification); a P1 PHI exposure invokes
  both. Confirm any My Health Record interaction (its own regime).

## 6. Verification (tested, not assumed)
A smoke/access suite asserts: warehouse/lakehouse readable; control SQL reachable + meta/log seeded; own-domain KV
get; cross-domain KV 403; Entra group assignment; non-prod stores carry no prod-PHI markers. Run after every rebuild.
