# 06 - CI/CD, repos, and the operating model

The repository topology, the pipelines + gates, and the human/AI operating model. This is how the platform is
delivered safely and reproducibly, with the owner in control of every apply.

## 1. Repository topology (split by lifecycle + deploy mechanism, not by file type)
A Fabric workspace's Git integration binds to one repo + one branch + one folder, so notebooks and pipelines that
share a workspace must share a repo; things that deploy differently are separate repos. **One ADO project, five repos:**
| Repo | Purpose | Deploy mechanism |
|---|---|---|
| docs | gameplan, ADRs, runbooks, FinOps/risk registers, per-phase docs + DSDs | review/PR |
| platform | infra (Terraform) + framework (fabric-items, control-plane DDL, the wheel) + devops + tests | Terraform + Fabric Git + CI/CD (path-filtered) |
| sql | warehouse serving only (the exception path) | sqlpackage / DACPAC |
| dataflows | per-flow manifests + bronze/silver/gold DDL + notebooks (selective CI by dag_uid) | dataflows CI/CD |
| bi | semantic models + reports (PBIP), selective by domain+item | Fabric Deployment Pipelines |
No repo commits the operating-rules file or local environment-identity file (both gitignored, local only).

## 2. Branch protection + promotion
main (-> Prod): PR + 2 approvals, required checks, no direct push. uat: PR + 1 approval + integration. dev:
PR + 1 approval + lint. feature/* -> dev only. Promote feature->dev->uat->main with gates; main->prod is manual.
Rollback: git revert + redeploy; Terraform state revert; Delta time-travel for data.

## 3. Pipelines + gates (single-purpose, generated-artifact-aware)
- **PR validation** (per repo): lint, unit, **manifest validate**, **no-committed-seeds** gate, **DDL-drift**
  gate (committed DDL == manifest), project-isolation guard (one project per PR), platform-boundary guard,
  typography gate (no em dashes/ambiguity), T-SQL parse of the *rendered* seeds/views (non-vacuous).
- **dataflows-build-deploy** (project_code, env): validate -> render+apply meta via `load_metadata` -> deploy
  notebooks/DDL (DDL deploy named-skips when there is no committed DDL - tables are Spark-created).
- **dataflows-run**: trigger the generic runner; DQ gate; **PublishViews** stage publishes the generated views to
  the lakehouse SQL endpoint AFTER the run creates the tables (a view cannot precede its table).
- **platform-build-wheel / platform-release**: build + unit-test the wheel (Spark tests importorskip; pyyaml
  installed so manifest/contract tests run; wheel version single-sourced + gated), publish to the Fabric Environment.
- **infra-deploy / ops-environment-rebuild**: terraform plan/apply; named-skip until backends/subscription exist;
  force UTF-8 so terraform diagnostics never crash a step.
- **finops-cost-report**: produce the report artifact; auto-PR to docs is best-effort (never fails the run).
- Auth: OIDC service connection + az tokens (zero SP secrets). Env values via ADO variable groups, never hardcoded.

## 4. Reproducibility (greenfield + rebuild)
Repos are the source of truth; rebuild by running the pipelines. Account-specific values live in ONE local,
gitignored `config/<target>.local.json` (emails/IDs/KV names; never secret values). Promotion trial -> trial ->
dev/test/prod is config-only (`capacity_mode` + env tfvars). The trial capacity is the one manual pre-step (Fabric
trials cannot be Terraformed); paid envs are created by the apply. Nothing is bundled/zipped - the package IS the
repos + the local config. The deploy driver ASKS before syncing to Azure DevOps.

## 5. The operating model (human + AI - BINDING)
- **No manual metadata** - all `meta.*` registration is via CI/CD (the loader), never a hand-run SQL write.
- **No AI footprint** - no AI commit trailer, no AI-named cloud identity, no "generated by" markers; writing reads
  as the team's own.
- **All cloud access via the owner's identity, read-only for assistance.** There is no separate assistant principal.
- **AI prepares LOCALLY; the OWNER reviews, syncs, and RUNS pipelines.** The assistant does not trigger CI runs.
  Every proposed change ships as a numbered guideline; the human applies it.
- **Errors fixed locally from an owner-granted read; the owner re-syncs.** This is the loop from Phase 1 onward.
- The disposable pilot may use a faster "apply live under the owner's token" mode; real environments do not.

## 6. CI capacity is a prerequisite
Pipelines need agent capacity. Microsoft-hosted free minutes are limited and will run out. Before relying on CI,
arrange one of: a self-hosted agent (free, immediate, `pool: {name: Default}`), the free parallelism grant
(request form), or paid parallel jobs. Without capacity, no fix can be run-verified.
